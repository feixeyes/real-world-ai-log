# 我是如何把工作经验，沉淀成 Codex CLI Skills 的

**中心问题**：从临时对话到系统协作，中间发生了什么？

**结构类型**：对比反思型（Before → After → Insight）

**目标读者**：
- 正在使用 Codex CLI 或其他 Coding Agent 的人
- 发现 AI 协作效果不够理想，想优化协作方式的人
- 对如何将个人工作经验系统化感兴趣的人

**字数范围**：1800-2200 字

---

## 大纲结构

### 1. 一开始的体验：不够"惊艳"

**核心要点**：
- 最近一段时间持续使用 Codex CLI 写代码
- 一开始的体验并不算"惊艳"
- 逐渐意识到：真正影响效果的，不是模型本身

**写作提示**：
- 开门见山，诚实描述初期体验
- 不夸大也不贬低，保持客观
- 为后续的"转折"做铺垫

---

### 2. 转折点：问题不在模型，而在协作方式

**核心要点**：
- 在多次使用后，逐渐形成了一套相对稳定的工作流程
- 这套流程包括：
  - 先让 Codex 充分理解当前项目
  - 再分析新功能或 issue 的影响范围
  - 制定整体计划
  - 我确认后，再拆解成小 task 执行
  - 每个 task 完成后，进行充分检查和测试
- 核心问题：我几乎每次都要把这套要求重新输入一遍
- 这既低效，也不利于持续优化

**写作提示**：
- 强调"逐渐形成"的过程感
- 列出具体的工作流程步骤，让读者能理解
- 点出痛点：重复输入

---

### 3. Before：重复输入的，其实不是 Prompt，而是"方法"

**核心要点**：
- 回过头看，我发现自己反复输入的，并不是某几句固定的 Prompt
- 而是一整套隐含的工作方法：
  - 不要一上来就写代码
  - 先理解项目上下文
  - 明确改动影响范围
  - 有计划、有节奏地推进
  - 每一步都可检查、可回滚
- 这些东西，对人来说是"常识"
- 但对 Codex 来说，如果不明确说出来，它是不会默认遵守的
- 问题：能不能把这些要求，变成 Codex 的"默认行为"？

**写作提示**：
- 强调"方法"而不是"Prompt"
- 用列表清晰呈现工作方法的各个层面
- 提出核心问题，引出 Skills

---

### 4. After：把工作流程，沉淀成 Skills

**核心要点**：
- 后来我给 Codex CLI 增加了两个自定义 skills：
  - add-feature
  - fix-issue
- 它们并不复杂，但各自都内置了一套明确的协作流程
- 以 add-feature 为例，大致包含几个阶段：
  1. 先深入理解当前项目结构和已有逻辑
  2. 分析新增功能可能涉及的模块和影响点
  3. 给出一个完整的实现计划
  4. 等我确认后，再把计划拆成多个小 task
  5. 每个 task 执行完成后，进行检查和测试
- 这样一来，我不再需要反复解释"我希望你怎么做"
- 而只需要告诉 Codex：现在是 add-feature 场景

**写作提示**：
- 给出具体的 Skills 名称和结构
- 详细展开其中一个 Skill 的流程（add-feature）
- 强调"场景化"的价值
- 对比 Before：不需要反复解释

---

### 5. 真正的变化：从"用 AI"到"训练协作方式"

**核心要点**：
- 一开始以为，Skills 的最大价值是省时间
- 后来发现，更重要的是另一点：它们是可以被持续优化的
- 比如：
  - 某次发现拆 task 拆得不够细
  - 或者测试阶段考虑不充分
  - 或者某类项目需要额外的检查步骤
- 我不会在 Prompt 里临时补一句
- 而是直接去调整 Skill 本身
- 久而久之，这些 Skills 就变成了：我个人工作习惯的外化版本
- 而 Codex，也越来越像一个熟悉我做事方式的搭档

**写作提示**：
- 强调"可迭代"比"省时间"更重要
- 用具体例子说明如何优化
- 提出"外化版本"和"搭档"的比喻
- 强调长期价值

---

### 6. 持续演进与写在最后

**核心要点**：
- 回头看这个过程，我最大的一个感受是：
  - 使用 AI 的关键，不是一次性写出"完美 Prompt"
  - 而是不断把隐性的判断和流程，显性化、结构化
- Skills 的意义，也不只是 Codex CLI 的一个功能
- 而是一种思路：
  - 把重复决策变成默认行为
  - 把个人经验沉淀为系统能力
  - 把临时对话，变成可复用协作模式
- 这件事一旦开始做，就会形成正反馈
- 对我来说，AI 实践记录的价值，就在于这些细节
- 不是模型有多强，而是我如何一步步，把 AI 真正嵌进自己的工作方式里
- 接下来，我也会持续记录这些 Skills 的演进过程
- 如果你也在使用 Codex 或其他 Coding Agent，也许可以从问自己一个问题开始：
  - 你现在反复输入的那些话，有没有可能，被沉淀成一个长期可复用的能力？

**写作提示**：
- 总结核心洞察：显性化、结构化
- 提出 Skills 的更广义价值
- 强调"正反馈"和持续演进
- 回归项目主题："AI 实践记录"的意义
- 以问题结尾，引发读者思考
- 保持开放和邀请的语气

---

## 写作约束

- **语气**：诚实、反思、第一人称
- **段落**：短到中等，避免长篇理论
- **避免**：
  - 教程式命令
  - 夸张的 AI 能力声明
  - 把自己描述成专家
  - 过度技术化的描述
- **优先**：
  - 具体的场景和例子
  - 真实的感受和发现
  - 可复用的思考路径
- **用词**：避免"应该"、"必须"、"最佳实践"
