# 我用 AI Coding Agent 做了三个"没用"的小项目

## 文章核心

**中心问题**：为什么"没用的小项目"反而是学习 AI Coding Agent 的最佳入口？

**目标读者**：想学习 AI Coding Agent 但不知道从哪里开始的人，或者觉得要做"有用的项目"才值得动手的人

**核心观点**：
- 学习新工具最好的方式不是做"正经项目"，而是先做一些"没用但好玩"的东西
- "写崩"是学习过程的一部分，反而能让你更真实地理解 AI Coding 的边界
- 心理门槛低、反馈快、容错空间大的小项目，能让你持续保持兴趣

---

## 大纲结构

### 1. 开头：为什么我不推荐一上来就做"正经项目"

**要点**：
- 很多人学习 AI Coding Agent 的典型心态：觉得要做"有用的东西"才值得
- 这种心态带来的问题：心理负担重、容错率低、容易放弃
- 我的做法：先做了几个"没什么用"的小项目
- 意外收获：反而在这些"没用"的项目中，建立了对 AI Coding 的真实感受

**语气**：轻松、坦诚，承认自己一开始也有这种想法

---

### 2. 项目 1：贪吃蛇游戏

**为什么选它**：
- 规则简单、功能明确
- 不需要复杂的架构设计
- 写崩了也不心疼

**做的过程**：
- 用什么工具（Cursor / Claude Code / Codex）
- 大概的开发流程（从描述需求到看到效果）
- AI 做得好的地方（比如快速搭建基础框架）
- AI 做得不好的地方（比如某些逻辑细节需要反复调整）

**"写崩"的经历**：
- 具体哪里崩了？（比如碰撞检测有 bug、游戏速度控制不对等）
- 怎么处理的？（重新描述需求、手动调试、还是干脆重来？）

**学到了什么**：
- 对 AI Coding 有了第一次直观感受
- 理解了 AI 擅长什么、不擅长什么
- 发现"写崩"并不可怕，反而是理解工具的过程

---

### 3. 项目 2：玩游戏背单词 web 项目

**为什么做这个**：
- 从纯游戏到"稍微有点用"的过渡
- 想尝试结合游戏化和实用功能

**做的过程**：
- 用什么工具
- 项目的核心功能设计（比如怎么把背单词和游戏结合？）
- AI 在这个项目中的表现
- 遇到的问题和调整

**"写崩"的经历**（如果有）：
- 哪些部分没按预期工作？
- 为什么崩？是需求描述不清，还是 AI 理解偏了？

**学到了什么**：
- 对 AI Coding Agent 的"对话能力"有了更深理解
- 开始摸索如何更好地描述需求
- 体会到大模型的强大（比如快速实现想法）

---

### 4. 项目 3：个人日程管理 / 简单操控小车游戏

**这一部分可以选择其中一个重点写，或者两个都简略提及**

**如果选个人日程管理**：
- 从"纯好玩"到"真正想用"的转变
- 实用项目的复杂度提升（比如数据存储、状态管理等）
- AI 在这种稍复杂项目中的表现
- "写崩"后的思考：是继续修，还是重构，还是放弃？

**如果选操控小车游戏**：
- 尝试更复杂的交互逻辑
- AI 在物理模拟、动画控制等方面的表现
- 折腾过程中的乐趣和挫折

**学到了什么**：
- 对 AI Coding 的边界有了更清晰的认知
- 开始形成自己的协作方式（比如怎么拆解需求、怎么验证结果）
- 意识到"写崩"不是终点，而是迭代的一部分

---

### 5. 复盘：这些"没用"的项目，到底有什么用？

**核心观点**：
- 它们降低了心理门槛：不用担心"做不好怎么办"，反正本来就是玩
- 它们提供了快速反馈：写几行代码就能看到效果，持续的正反馈让人愿意继续
- 它们允许试错：写崩了不心疼，反而能更真实地理解 AI 的能力和局限
- 它们培养了协作直觉：通过反复对话，逐渐摸索出怎么和 AI 更好地配合

**对比**：
- 如果一开始就做"正经项目"会怎样？（压力大、容错率低、容易因为一次失败就放弃）

**结论**：
- "没用"不是浪费时间，而是最低成本的学习方式
- 真正的学习不是"一次做对"，而是"敢于试错"

---

### 6. 写在最后：给想上手的人

**建议**：
- 不要一开始就想着做"有用的东西"
- 选一个你感兴趣的小项目（游戏、小工具、交互页面都行）
- 不要怕"写崩"，崩了再来，或者换一个
- 关注过程中的感受：AI 哪里做得好，哪里做得不好，你是怎么配合它的

**结尾语气**：
- 轻松、鼓励性的
- 强调"玩"的心态比"学"的心态更重要
- 用大模型，做真实的事（哪怕是"真实地折腾"）

---

## 写作注意事项

- **语气**：第一人称，轻松、坦诚，不装专家
- **具体性**：尽量用具体的细节（用了什么工具、遇到什么问题、怎么解决的）
- **真实性**：不回避"写崩"的经历，反而把它当作重要的学习素材
- **节奏**：短到中等段落，不要长篇大论
- **边界**：不夸大 AI 能力，也不贬低它，保持客观的实践者视角
